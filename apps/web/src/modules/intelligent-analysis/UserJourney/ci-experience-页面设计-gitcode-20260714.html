附录 B · 失败归因算法(两道守卫 + 判别规则)
  1. 定位首因 job:FAILED run 内,取 status=='FAILED' 且失败 step 结束最早的 leaf job(排除 message==null 的父级包装节点);fail-fast 保证首因即根因位置。
  2. 取信号:优先 job.message;为 null 则遍历 steps[] 取 FAILED step 的 name。不看日志也能定位到步骤级。
  3. 广度守卫:同一 run 内失败的独立 UT 域/编译变体 ≥3,或同一失败 step 短窗内横跨 ≥5 个 PR → 判「疑似系统性·暂缓代码红」,压住机器人代码红回帖,转基础设施 triage(实证:07-14 单 run 17/18 个 UT 域同挂)。
  4. 上游产物守卫:download 失败先查上游产物是否成功产出——上游成功才判 OBS(CI 红),上游失败归上游因;image 阶段按 step 拆(revise_image/资源错→CI 红;get_pr 解析失败→待判定,畸形 PR 可致其失败)。
  5. 归类(按序命中即止):start_time==null 或 message 含「资源/内部错误/执行机未注册」或首因落 image(非 get_pr 解析类)→ CI 自身红;首因落 compile/ut/staticcheck/check_pr 且有带名步骤 → 代码红;首因 resource=self-hosted → 真机待判定。

禁止把「未归因/待判定」塞进实桶充数。CANCELED 不计失败(含 by-design 抢占);INIT 是 fail-fast 正常跳过。叶子级哑信号实测=0;父节点不透传(104 处)列可读性改进项。
