# 开发者体验模块前端架构与路由迁移总体设计方案

> 文档状态：方案稿  
> 更新日期：2026-07-17  
> 适用范围：`compass-web/apps/web` 的社区入门体验、Issue 贡献体验、CI 体验及体验总览  
> 核心约束：兼容全部既有入口；原社区入门体验入口必须重定向到新的 canonical URL

## 1. 结论摘要

建议在 `intelligent-analysis` 下新增统一的 `DeveloperExperience` 体验域，三类体验报告位于同一层级：

```text
/intelligent-analysis/experience/community-onboarding
/intelligent-analysis/experience/issue-contribution
/intelligent-analysis/experience/ci-experience
```

现有 `UserJourney` 已经深度绑定社区入门体验的数据结构、报告组件、总览 API、仓库管理和任务管理，不宜直接继续承载 Issue/CI 业务。首期不搬迁现有实现，而是：

1. 新建 `DeveloperExperience` 作为公共路由、导航、页面壳和未来总览组合层。
2. 将现有 `UserJourney` 作为 `CommunityOnboarding` 的 legacy adapter 复用。
3. Issue、CI 分别建立独立的类型、组件、fixture、hook 和 API 边界。
4. 对已有等价 canonical 页的四个旧报告/看板入口使用 307 临时重定向，并完整保留原 query；社区运营页和 `/user-journey` 暂时保持旧路径 200。
5. 原社区入门体验入口执行以下重定向：

```text
/intelligent-analysis/community-experience
    ↓ 307
/intelligent-analysis/experience/community-onboarding
```

6. 当前总览只接入社区入门体验；Issue/CI 通过配置保持 `showInOverview: false`，后续再逐个接入。

## 2. 设计目标与非目标

### 2.1 设计目标

- 三类体验模块拥有稳定、可分享、可刷新、支持浏览器前进/后退的 URL。
- 旧书签、旧详情链接、外部系统入口及已有 query 参数继续有效。
- 模块级导航、报告对象选择、页内导航三层职责清晰。
- CI 页面按 2026-07-14 HTML 设计稿 React 化。
- Issue 在正式页面设计提供前，先稳定路由、页面边界和空态，不虚构业务页面。
- 前端先使用强类型 fixture，后端 API 上线后只替换模块内 loader。
- 总览未来可组合三个模块，但不构造不成立的跨模块总分。

### 2.2 非目标

- 首期不整体搬迁或重写现有 `UserJourney`。
- 首期不改写社区入门体验的评分和报告数据结构。
- 首期不让 Issue/CI 进入现有总览统计。
- 首期不为 Issue 设计未经确认的最终视觉稿。
- 不把 CI 六灯、Issue A–E 维度和社区旅程步骤抽象成同一个万能报告组件。
- 不在前端新增第二套评分、灯态或阈值计算规则。

## 3. 当前实现与主要问题

### 3.1 当前页面路由

现有社区入门体验同时维护无组织和带组织的两套路由：

```text
/intelligent-analysis/community-experience
/intelligent-analysis/:org/community-experience

/intelligent-analysis/community-experience/overview
/intelligent-analysis/:org/community-experience/overview

/intelligent-analysis/community-experience/repo-management
/intelligent-analysis/:org/community-experience/repo-management

/intelligent-analysis/community-experience/task-management
/intelligent-analysis/:org/community-experience/task-management
```

另有旧入口：

```text
/intelligent-analysis/user-journey
```

该入口与 `community-experience` 并非简单别名：它额外包含全局 Header、`AuthRequire allowedRoleLevels={[3, 7]}`，默认显示 developer type/journey mode 控件，且不处理 `soc=A5` 空态，因此不能在未统一权限语义和页面能力前直接重定向。

### 3.2 当前代码边界

- `UserJourney/index.tsx` 是社区入门体验报告容器，解析 `project`、`sig`、痛点定位和预览参数。
- `UserJourney/components/PageHeader.tsx` 是社区报告操作栏，强依赖组织、SIG、项目、硬件、版本和对比逻辑，不是模块级导航。
- `OverviewDashboard` 目前只统计社区入门体验，并使用社区专属的 score、pain point、close rate 等模型。
- `rawData/apiClient.ts` 的 API 前缀仍是 `/user-journey-api/user-journey`，不应继续追加 Issue/CI DTO。
- 仓库内多处直接拼接 `/intelligent-analysis/community-experience`，存在组织上下文丢失和未来迁移成本。
- 现有页面多层使用 `h-screen` 与 `overflow-y-auto`，新增公共导航后容易形成双滚动。

### 3.3 设计稿代际裁决

CI 的 2026-07-09 草案仍包含 CDX 总分、A–F 等级和 C0–C4 旅程；2026-07-14 HTML 已明确改为：

- 不打总分、不评等级；
- 六灯行动板；
- 失败归因分流；
- 轨 A（CI 基础设施）和轨 B（贡献者/运营）；
- REC 改进闭环、缺口、趋势和附录。

因此，CI 页面应以 `ci-experience-页面设计-gitcode-20260714.html` 为当前页面权威，旧草案只作为领域背景，不作为前端页面结构权威。

Issue 指标细则则明确存在两个不同对象：

1. 仓库周度体验报告；
2. 单条 Issue 体验详情。

两者需要独立页面边界，不能塞进单个大页面或只使用弹窗承载。

## 4. 目标信息架构

### 4.1 一级：体验总览与模块导航

公共体验导航由一个独立的总览入口和三个模块 Tab 组成：

```text
[总览看板]  [社区入门体验] [Issue 贡献体验] [CI 体验]
```

- “总览看板”与三个模块 Tab 在信息架构上分离。
- 三个模块 Tab 的激活状态完全来自 URL。
- 模块 Tab 不使用组件本地 state 模拟路由。
- 当前总览只展示社区入门体验数据，但三类报告均可从导航直接访问。

### 4.2 二级：报告对象上下文

由各模块自己提供：

- 社区入门体验：组织、SIG、项目、硬件、版本、对比项目。
- Issue：平台、组织、仓库、周周期。
- CI：平台、组织、仓库、workflow、周周期、样本与置信度。

公共 Shell 不理解这些领域字段，只负责承载模块组件。

### 4.3 三级：报告页内导航

长报告使用 hash 作为稳定章节定位：

```text
#verdict
#attribution
#track-a
#track-b
#improvement-loop
#data-gaps
#trend
#appendix
```

页内导航不使用 query，也不触发报告数据重新请求。

## 5. Canonical 路由设计

### 5.1 总览

```text
/intelligent-analysis/experience/overview
/intelligent-analysis/:org/experience/overview
```

该路由区别于已有的 `/intelligent-analysis/overview`；后者是原智能分析总览，不属于本方案的开发者体验总览。

### 5.2 社区入门体验

```text
/intelligent-analysis/experience/community-onboarding
/intelligent-analysis/:org/experience/community-onboarding
```

`repo-management` 和 `task-management` 是社区运营页面，不属于三类体验报告的公共信息架构。首期继续使用原路径并返回 200，不创建未经确认的新 canonical 管理路径；如果后续单独完成运营信息架构设计，再为它们增加新入口和重定向。

### 5.3 Issue 贡献体验

仓库周报：

```text
/intelligent-analysis/experience/issue-contribution
/intelligent-analysis/:org/experience/issue-contribution
```

单条 Issue 详情：

```text
/intelligent-analysis/experience/issue-contribution/issues/:issueId
/intelligent-analysis/:org/experience/issue-contribution/issues/:issueId
```

`issueId` 应使用后端稳定标识；展示用 Issue number 不应被假设为跨仓库全局唯一。

### 5.4 CI 体验

```text
/intelligent-analysis/experience/ci-experience
/intelligent-analysis/:org/experience/ci-experience
```

CI 首期只有报告页，不创建伪总览、仓库管理或任务管理子页面。

## 6. 旧路由兼容与重定向

### 6.1 兼容原则

旧地址兼容与模块切换是两种不同场景：

#### 旧地址重定向

- 旧地址和新地址表达同一个页面资源。
- 必须保留全部已知和未知 query 参数，包括按原顺序重复的 `project`、`sig`、`preview`、`reviewId`、`adminToken`、`focusTaskId`、`painId`、`focusPainIndex`、`autoOpenPain`、`soc` 等；不得转 CSV、去重或白名单过滤。
- 服务端看不到 hash。重定向目的地址不写入新 fragment，由浏览器继承原 fragment，并用真实浏览器 E2E 验证部署链路。
- 迁移期使用 307，不使用 301/308，避免浏览器和代理永久缓存错误规则。

#### 模块切换

- 表达从一个业务模块进入另一个业务模块。
- 只能白名单保留跨模块有效上下文。
- 必须清除模块专属 query、筛选和 hash。
- 特别禁止将 `adminToken` 带入 Issue/CI URL。

### 6.2 重定向矩阵

| 旧 URL | 新 canonical URL | 首期策略 |
| --- | --- | --- |
| `/intelligent-analysis/community-experience` | `/intelligent-analysis/experience/community-onboarding` | 307，立即启用 |
| `/intelligent-analysis/:org/community-experience` | `/intelligent-analysis/:org/experience/community-onboarding` | 307，立即启用 |
| `/intelligent-analysis/community-experience/overview` | `/intelligent-analysis/experience/overview` | 307，立即启用 |
| `/intelligent-analysis/:org/community-experience/overview` | `/intelligent-analysis/:org/experience/overview` | 307，立即启用 |
| `/intelligent-analysis/community-experience/repo-management` | 暂无新 canonical | 保持旧路径 200，不重定向 |
| `/intelligent-analysis/:org/community-experience/repo-management` | 暂无新 canonical | 保持旧路径 200，不重定向 |
| `/intelligent-analysis/community-experience/task-management` | 暂无新 canonical | 保持旧路径 200，不重定向 |
| `/intelligent-analysis/:org/community-experience/task-management` | 暂无新 canonical | 保持旧路径 200，不重定向 |
| `/intelligent-analysis/user-journey` | 待权限统一后指向 community onboarding | 首期保留原页面，不立即重定向 |

### 6.3 为什么首期使用 307

- 保证迁移可回滚。
- 避免浏览器永久缓存 canonical 决策。
- 保留原请求方法语义。
- 便于在日志中观察旧地址使用量，再决定是否升级为 308。

### 6.4 `next.config.js` 重定向示例

以下仅表示方案结构，实施时应加入现有 `redirects()` 返回数组，并优先放置更具体的规则。只有 destination 与旧页面等价且在同一版本上线后，才启用相应 307：

```js
async redirects() {
  return [
    {
      source: '/intelligent-analysis/community-experience/overview',
      destination: '/intelligent-analysis/experience/overview',
      permanent: false,
    },
    {
      source: '/intelligent-analysis/:org/community-experience/overview',
      destination: '/intelligent-analysis/:org/experience/overview',
      permanent: false,
    },
    {
      source: '/intelligent-analysis/community-experience',
      destination: '/intelligent-analysis/experience/community-onboarding',
      permanent: false,
    },
    {
      source: '/intelligent-analysis/:org/community-experience',
      destination:
        '/intelligent-analysis/:org/experience/community-onboarding',
      permanent: false,
    },
  ];
}
```

Next.js 对 `permanent: false` 使用临时重定向。验收时必须实际验证：

- 单个 query；
- 重复 `project` query；
- URL 编码后的 `org` 和项目值；
- hash；
- `soc=A5`；
- preview URL。

服务端无法读取 fragment。如果真实浏览器 E2E 发现目标浏览器或跳转链丢失 hash，应在旧页面使用能读取 `window.location.hash` 的最小客户端兼容组件执行 `router.replace()`；不得声称服务端 307 能显式复制 hash，也不得先渲染完整旧报告再跳转，以免重复请求和重复上报埋点。

以下路径不是开发者体验旧路由，不得纳入本次重定向：

```text
/intelligent-analysis/overview
/intelligent-analysis/overview/:project
/user-journey-api/:path*
/data/intelligent-analysis/user-journey/:path*
```

前两个属于另一套智能分析总览；后两个分别是开发环境 API rewrite 和静态数据地址。

### 6.5 旧页面文件保留周期

即使 `next.config.js` 已配置重定向，首期也不立即删除旧 `pages` 文件：

1. 保留至少一个完整发布周期作为回滚保险。
2. 观察旧 URL 请求量、外部引用、监控和错误率。
3. 确认 canonical 页面权限、query、空态和埋点全部一致后再删除薄页面。
4. 仓库内不得继续新增旧 URL 字符串。

## 7. URL 状态规范

### 7.1 Path、query、hash 和本地状态

| 状态类型 | 放置位置 | 示例 |
| --- | --- | --- |
| 模块身份 | Path | `community-onboarding`、`issue-contribution`、`ci-experience` |
| 组织上下文 | Path | `/:org/experience/...` |
| 报告对象选择 | Query | `repo`、`period`、`workflow`、`project` |
| 可分享的列表钻取 | Query | `dimension`、`risk`、`confidence`、`page` |
| 报告章节 | Hash | `#attribution`、`#track-a` |
| 折叠、非关键排序、临时弹窗 | 本地 state | `openSection`、`modalOpen` |

### 7.2 各模块 query

#### 社区入门体验

继续兼容现有参数：

```text
project=<fileKey>&project=<compareFileKey>
sig=<sig>
focusTaskId=<taskId>
painId=<painId>
focusPainIndex=<index>
autoOpenPain=true
preview=true
reviewId=<reviewId>
adminToken=<token>
soc=A5
```

社区总览旧地址还需原样兼容以下 query：

```text
expandAllTeams=true
captureMode=true
includeCommonIssues=true
```

#### Issue 贡献体验

```text
platform=gitcode
repo=runtime
period=2026W29
dimension=response
risk=severe
confidence=low
reviewStatus=pending
page=1
```

#### CI 体验

```text
platform=gitcode
repo=runtime
period=2026W29
workflow=runtime_action
```

首期“观察窗代周期”的 `windowStart`、`windowEnd`、`windowType=baseline` 应放在报告响应的 Meta 中，不固化为 URL 协议。

### 7.3 模块切换保留规则

| 来源 → 目标 | 保留 | 清除 |
| --- | --- | --- |
| 任意模块 → 任意模块 | `org`；目标支持时保留 `platform`、`repo` | 原页面 hash、列表筛选、详情定位 |
| Issue ↔ CI | `org`、`platform`、`repo`、`period` | `workflow` 或 Issue 专属筛选 |
| Community → Issue/CI | 可解析时映射 `org`、`repo` | `project`、`sig`、痛点参数、preview 参数 |
| Issue/CI → Community | `org`；可解析时映射目标项目 | `period`、`workflow`、Issue 筛选 |

模块切换后，如果目标模块不存在相同 scope 的报告，显示“当前仓库/周期暂无报告”，不得静默跳到另一个仓库或最新周期。

## 8. Pages Router 文件设计

```text
src/pages/intelligent-analysis/
├─ experience/
│  ├─ overview.tsx
│  ├─ [experienceType].tsx
│  └─ issue-contribution/
│     └─ issues/
│        └─ [issueId].tsx
│
└─ [org]/experience/
   ├─ overview.tsx
   ├─ [experienceType].tsx
   └─ issue-contribution/
      └─ issues/
         └─ [issueId].tsx
```

原则：

- `pages` 文件只解析路由、国际化、权限和 scope，保持为薄入口。
- 无组织和带组织页面复用同一个 `ExperienceReportRoute`。
- `[experienceType]` 必须通过 registry 白名单验证；非法类型返回 404。
- 组件映射必须显式声明，避免 `import(\`./${type}\`)` 造成打包边界不确定。
- 静态 `overview.tsx` 与专属子路径优先于动态 `[experienceType].tsx`。

## 10. Experience Shell 设计

### 10.1 页面结构

```text
ExperienceShell
├─ ExperienceTopNavigation
│  ├─ OverviewLink
│  └─ ExperienceModuleTabs
└─ ScrollContainer
   └─ ModulePage
      ├─ ModuleContextBar
      ├─ InPageNav（可选）
      └─ ReportBody
```

### 10.2 滚动与 Sticky

- Shell 是唯一拥有纵向滚动的容器。
- 子页面从 `h-screen` 调整为 `min-h-full` 或内容自然高度。
- 模块导航固定在 ScrollContainer 之外。
- CI 页内导航在 ScrollContainer 内 `sticky top-0`。
- 不允许 route wrapper、Shell、报告正文同时拥有 `overflow-y-auto`。

### 10.3 `soc=A5` 空态

当前社区入口在挂载 `UserJourney` 前直接返回整页空态。迁移后应把 `soc=A5` 判断移动到 `CommunityOnboarding` 子页面内部：

```text
ExperienceShell
├─ 三模块导航仍然可见
└─ CommunityOnboardingEmptyState
```

否则 A5 用户无法从社区空态切换到 Issue/CI。

### 10.4 权限

- Shell 只负责布局和导航，不统一放宽或收紧权限。
- 报告、总览、仓库管理、任务管理分别保留现有权限语义。
- `/intelligent-analysis/user-journey` 在权限逻辑审计完成前不重定向。
- 后续如统一鉴权，应先形成明确的角色矩阵，再调整入口。

```tsx
    />
```

需要同时处理：

- 返回总览地址改由 `routes.ts` 生成；
- 现有 22 处左右的社区 URL 硬编码逐步替换为 route builder；
- 带组织路由生成的所有详情链接必须保留 `org`。

首期不移动 `OverviewDashboard`、`RepoManagementPage`、`TaskManagementPage` 和社区专属 components。

四个社区运营入口继续保持旧路径 200。它们内部的跳转也应改用集中 route builder，以修复 `org` 丢失，但 route builder 在当前阶段仍生成原运营 URL。

## 12. CI 体验页面设计

### 12.1 页面定位

CI 页采用“工业化运行诊断台”的视觉方向：高信息密度、清晰证据层级、克制的状态色和强可扫描性。公共 Shell 保证品牌与导航一致，报告正文不强行套用社区入门体验的渐变旅程卡样式。

### 12.2 页面结构

按 2026-07-14 HTML 顺序拆分：

```text
CiReportPage
├─ CiReportMetaBar
├─ CiDisclosureBanners
├─ CiInPageNav
├─ CiVerdictLights
├─ CiAttributionPanel
│  └─ ContributorLookupTable
├─ CiInfrastructureTrack
├─ CiContributorTrack
│  └─ CiRunDurationHistogram
├─ CiImprovementLoop
├─ CiDataGaps
├─ CiTrendSection
└─ CiMetricAppendix
```

### 12.3 数据模型

```ts
type CiReportScope = {
  platform: string;
  org?: string;
  repo: string;
  workflow?: string;
  period: string;
};
```

- 将 HTML 中唯一的 `DATA` 常量转成强类型 fixture。
- 仓库切换必须更新 URL，不再只调用本地 `switchRepo()`。
- JSX 中不得散落报告数值和阈值。
- 灯态、归因和建议由报告数据提供，前端仅渲染。
- HTML 中包含 `<b>` 的字符串应转为结构化字段，不直接大范围使用 `dangerouslySetInnerHTML`。

### 12.4 当前权威口径

- 不展示 CI 总分。
- 不展示 A–F 等级。
- 六灯之间不汇总、不加权、不排名。
- 同源灯必须显式标注，避免重复解释同一故障事件。
- 失败归因决定 owner 和后续操作。
- 缺失数据进入“诚实边界”，不使用占位分数。

## 13. Issue 贡献体验页面预留

### 13.1 首期状态

Issue 页面设计未正式提供，因此首期：

- 路由和模块 Tab 可用；
- 展示 `ReportComingSoon`；
- 保留当前 `org/platform/repo/period` 上下文；
- 不生成模拟得分、图表或代表性 Issue；
- 不进入总览。

### 13.2 仓库周报页面边界

根据指标文档预留：

```text
IssueRepositoryReportPage
├─ ReportMeta
├─ WeeklyConclusion
├─ CoverageAndPublicationStatus
├─ DimensionOverview（A–E）
├─ SampleFunnel
├─ PoolBreakdown（新增池/过程池/结果池）
├─ RiskAndReviewList
├─ FourWeekTrend

```

### 13.3 单条 Issue 详情边界

```text

├─ ScoreAndCoverage
├─ ObservedMilestoneTimeline
├─ DimensionAndMetricDetail
├─ ThreeLayerEvidence
├─ SevereIssueCaps
├─ SourceEventLinks
```

关键规则：

- 覆盖率不足 80% 时必须显示“阶段性”。
- 某维度无适用指标时不显示维度分，不能显示为 0。
- 有效样本 1–4 条时不发布正式周度等级。
- 没有有效样本时不发布体验分。
- 里程碑只展示实际发生事件，不绘制为强制线性流程。

## 14. 总览看板演进

### 14.1 当前阶段

新 canonical 总览：

```text
/intelligent-analysis/experience/overview
/intelligent-analysis/:org/experience/overview
```

当前只渲染：

```text

    └─ 现有 OverviewDashboard
```

模块配置：


### 14.2 后续阶段

总览逐步演进为组合器：

```text
[全部] [社区入门] [Issue] [CI]
```

每个模块 adapter 只输出跨模块可以合理比较的中性摘要：

```ts
type ExperienceOverviewSummary = {
  experienceType: ExperienceType;
  updatedAt: string | null;

    key: string;
```

- 将社区总分、Issue 得分和 CI 六灯合成为一个跨模块总分；
- 将 CI 灯态强行映射为 0–100 分；
- 将不同模块的缺失数据视为 0；
- 复用社区 `painPoints/closeRate/latestScore` 作为三模块公共 DTO。

## 15. 前端数据与 API 边界

### 15.1 模块内聚

每个新模块独立维护：

```text
types.ts   原始 DTO、ViewModel、状态枚举
api.ts     fetch 函数
hooks/     React Query 和 URL scope 解析
fixtures/  API 未上线前的强类型数据
```

不要向 `UserJourney/rawData/apiClient.ts` 继续追加 Issue/CI 接口。

### 15.2 React Query Key

```ts
[
  'developer-experience',
  experienceType,
  platform,
  org,
  repo,
  workflow,
  period,
]
```

必须包含 `experienceType` 和完整报告 scope，防止跨模块、跨组织或跨周期缓存串数据。

### 15.3 Fixture 到 API 的替换

```text
URL scope
   ↓
useCiReport / useIssueReport
   ↓
module loader
   ├─ Phase 1: typed fixture
   └─ Phase 2: HTTP API
   ↓
ViewModel
   ↓
React components
```

页面组件不感知数据来自 fixture 还是 API。

## 16. 路由工具设计

所有新旧链接统一从 `DeveloperExperience/routes.ts` 生成：

```ts
type ExperienceRouteScope = {
  org?: string;
  platform?: string;
  repo?: string;
  period?: string;
  workflow?: string;
};

buildExperienceOverviewHref(scope);
buildCommunityOnboardingHref(scope, communityQuery);
buildIssueContributionHref(scope, issueQuery);
buildIssueDetailHref(scope, issueId, issueQuery);
buildCiExperienceHref(scope, ciQuery);
buildLegacyCommunityRepoManagementHref(scope);
buildLegacyCommunityTaskManagementHref(scope);
```

要求：

- Path segment 统一 `encodeURIComponent`。
- query 使用 `URLSearchParams` 或等价的确定性序列化。
- 保留重复 `project` 参数能力。
- 不通过字符串 replace 从旧地址推导新地址。
- 所有模块 Tab 和详情链接都使用 route builder。
- 禁止新增硬编码 `/intelligent-analysis/community-experience`。

## 17. 分阶段实施计划

### Phase 0：路由地基

- 新增 `DeveloperExperience` 目录。
- 新增 module registry、route builder、Experience Shell。
- 新增无组织/带组织的报告动态入口和总览入口。
- 接入现有社区报告和总览 adapter。
- 配置社区 root、org root 和 overview 的 307 重定向。

验收：旧地址和新地址展示同一数据，query/hash 行为一致。

### Phase 1：CI 前端页面

- 将 HTML `DATA` 转为 typed fixture。
- 按章节拆分 React 组件。
- 仓库、周期和 workflow 切换写入 URL。
- 接入模块导航和页内 hash 导航。
- 不接总览。

验收：runtime/ops-nn 切换、刷新、分享和浏览器返回均正确；页面不出现 CI 总分。

### Phase 2：Issue 路由与页面边界

- 提供仓库周报和单 Issue 详情路由。
- 首期展示稳定 Coming Soon/No Report 空态。
- 页面设计确认后再实现正式 UI。
- 不接总览。

验收：路由可直达；没有虚构数据；上下文和空态准确。

### Phase 3：旧链接收口

- 将现有社区硬编码 URL 迁移到 `routes.ts`。
- 修复详情链接丢失 `org` 的问题。
- repo-management/task-management 继续使用旧路径 200，但统一由 legacy route builder 生成。
- 观察旧 URL 流量和错误率。

验收：代码中不再新增旧社区 URL；旧书签仍然有效。

### Phase 4：后端 API

- 分模块增加 CI/Issue API。
- fixture 与 API 共用 ViewModel adapter。
- Query Key 加入完整 scope。
- 完善 loading、error、partial、not-applicable 状态。

验收：切换数据源不修改主要页面组件。

### Phase 5：总览扩展

- 新增 Issue/CI overview adapter。
- 逐项将 `showInOverview` 改为 `true`。
- 添加“全部/社区入门/Issue/CI”筛选。
- 保持模块详情指标独立。

验收：总览不产生跨模块伪总分，缓存和 API 均包含模块类型。

### Phase 6：可选目录迁移

稳定后再评估是否将 `UserJourney` 移入：

```text
DeveloperExperience/CommunityOnboarding/
```

该步骤最后执行，不作为新增 Issue/CI 的前置条件。

## 18. 风险与控制措施

| 风险 | 控制措施 |
| --- | --- |
| 旧书签失效 | 四个等价入口使用 307 + query 原样传递 + hash 浏览器 E2E + 保留旧 pages 回滚周期 |
| 组织上下文丢失 | route builder 统一处理 `org`，禁止组件硬编码 URL |
| preview Token 泄漏到其他模块 | 模块切换使用 query 白名单，明确删除 `adminToken` |
| 双滚动、sticky 失效 | Shell 是唯一滚动容器，子页面移除 `h-screen` |
| A5 空态无法切模块 | 空态放在 Community adapter 内，Shell 始终渲染 |
| 权限意外变化 | Shell 不统一鉴权，`user-journey` 暂不重定向 |
| CI 页面使用过期口径 | 07-14 HTML 为页面权威，不实现旧草案总分 |
| Issue 页面过早定型 | 首期只做路由、空态和双页面边界 |
| 三模块类型互相污染 | DTO/API/hooks 模块内聚，只共享稳定基础能力 |
| 总览缓存串模块 | Query Key 和 API 参数加入 `experienceType` |
| 重定向循环 | canonical 页面不得反向生成旧 URL，增加自动化测试 |

## 20. 最终决策

本方案采用：

```text
DeveloperExperience 公共壳
    + CommunityOnboarding legacy adapter
    + IssueContribution 独立模块
    + CiExperience 独立模块
    + Overview adapter 组合机制
    + 旧路由 307 兼容层
```

最先落地的路由行为是：

```text
/intelligent-analysis/community-experience
    ↓ 307 Temporary Redirect（query 原样传递，hash 由浏览器继承并做 E2E）
/intelligent-analysis/experience/community-onboarding
```

这样能够在不重写现有社区入门体验的前提下，先完成三模块并列、CI 页面落地和 Issue 页面预留，同时为后续 API 与跨模块总览保留清晰、可演进的边界。
