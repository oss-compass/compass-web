# CI 体验报告重构方案（v0.2 草案）

> 2026-07-09 起草，同日按 **CI 专家评审意见**修订升 v0.2（收编清单见 §10.5）。基准：ISSUE 线设计权威（`../ISSUE/重构方案_Issue贡献体验报告_20260702.md` v1.1 + §16 v2.1 听众优先修订）与 07-09 依赖图诊断工具（`report/07-09_dep_graph.html`）。样例项目 NVIDIA/Megatron-LM（version 9aea，2026-06-01→06-26，N=61 run）。

## 0. 一句话定位

> CI 体验报告 = **开发者旅程中"提交之后等待机器裁决"环节的周期性回溯诊断**：在既有流水线诊断层（DAG + RCA，给 CI 维护者修问题用）之上，叠加一个开发者体验层（CDX 指标 + 旅程 + 决策物，回答"提交一个 PR 的遭遇如何、谁来改、改了有没有用"）。**两层一报告，不是替换而是叠加。**

## 1. 核心设计立场

1. **双听众、两层、一份报告，维护者视图先行**（用户裁决 + 专家评审 2026-07-09）：闭环改进体验的团队就是 CI 维护者，诊断层是资产而非旧物。报告主体顺序 = 诊断摘要（30 秒，两类听众共用）→ **维护者视图**（CI 自身的效率/稳定性/成本：定性阶段账单 → 两层细化 → 成本 → 失败归因分类 → 依赖图）→ **开发者体验视图**（旅程 C0–C4 + 痛点）→ 改进闭环。体验层的每个痛点必须能一键下钻到诊断层证据（job_id / 日志行 / file:line）。
2. **借形不借义**（沿 ISSUE §1）：从 SDX/IDX 借的仅是呈现形式——等级卡、旅程卡导航中枢、REC 卡、三色灯、置信度徽标、附录指标字典。CI 数据本体是 **run/job 遥测 + log agent 代读诊断**，不是体验模拟也不是问卷；附录须诚实声明"体验分是对遥测的换算与 LLM 日志代读，非开发者直接反馈"。
3. **决策物硬门**（沿 ISSUE v2.1 规则 1）：Top 痛点表与 REC 卡非空是出报告条件；建议必须从各 job RCA 中**去重收敛**，带稳定 ID、优先级、两级承接、状态字段。
4. **数据口径以 07-09 管道为准**（用户裁决 2026-07-09）：与 `report/_report.md`（06-27 代）的数字矛盾不再追溯，旧代际产物视为已废弃基线；本方案所有数值取自 07-09 嵌入数据。

## 2. 命名、阶段与框架挂载

### 2.1 两条正交的"阶段"轴（不合并）

- **工序阶段（维护者轴，专家 1.1）**：流水线按"做什么工作"定性分段，用于账单、两层细化与失败归因——**不用于体验评分**。
- **旅程阶段（开发者轴）C0–C4**：按"提交者的遭遇"分段，用于 CDX 评分。
- 07-09 的 gate/build/test/post 四类保留为**底层遥测 type**，报告主分类改用工序阶段。

**工序阶段分类法**（平台无关，五阶段 + 编排横切）：

| 代号 | 名称 | 判据 | Megatron-LM 映射 |
|---|---|---|---|
| ST | 静态检查 | 不执行被测代码的检查（lint/format/类型/许可证/DCO） | linting |
| BD | 编译打包 | 编译、构建镜像/多包联合包 | cicd-container-build |
| UT | 单元测试 | 进程内测试（原始用例 + 新增代码用例） | cicd-unit-tests-latest |
| FT | 功能级测试 | 整仓到真实/模拟环境的集成测试 | cicd-integration-tests-latest-h100 / gb200 |
| XR | 跨仓测试 | 触发下游/关联仓验证 | cicd-mbridge-testing |
| ORCH | 编排与治理（横切） | 身份/调度/占位/矩阵生成/汇总/通知/清理 | 其余 11+ 个轻量 job |

每个工序阶段支持**两层细化**（专家 1.2）：阶段行 → 展开 job 明细（job 名、一句话功能[取自 log 知识]、样本数、均时、失败率、判定）。

### 2.2 旅程阶段（开发者轴）

- **指标前缀**：`CDX_*`（CI Developer eXperience），与 SDX_* / IDX_* 平行。四类：`CDX_OBJ_*`（客观遥测，规则计算）、`CDX_DIAG_*`（诊断层产出，log agent 代读）、`CDX_REF_*`（参考量，仅展示不计分）、红旗规则（布尔，触发徽标不入均分）。

| 阶段ID | 名称 | 体验主体 | 阶段问题 |
|---|---|---|---|
| C0_ENTRY | 🚦 C0 进入与排队 | 提交者 | 提交后是否立刻被接住？ |
| C1_SIGNAL_LATENCY | ⏱️ C1 信号等待 | 提交者 | 多久拿到一个可行动的信号？快速失败还是深水区失败？ |
| C2_SIGNAL_QUALITY | 🚨 C2 信号质量 | 提交者 | 红灯可信吗——是我的代码问题还是环境/流程问题？ |
| C3_RECOVERY | 🔧 C3 恢复与修复 | 提交者+维护者 | 失败后能多快定位、能否本地复现、多快回到绿灯？ |
| C4_OUTCOME | ✅ C4 达成与稳定 | 提交者 | 最终能不能过？结果可预期吗？ |
| R_INFRA | 🏗️ 透镜 R 资源与基础设施 | 维护者 | runner/cache/下游调度是在助力还是拖累？（横切带，暂不入总分） |

- **透镜 R 处置**（对位 ISSUE 透镜 G）：基础设施性遭遇覆盖 run 占比 ≥30% 时升级为计分维度并首屏声明（阈值待校准）。
- **框架挂载：调取而非供数**：报告调取 v2.8 L3.D CI/CD 模型族已定义指标作为客观层对齐基准；附录 A 设"框架对应"列，Phase 1 逐行标注（本方案不编造具体框架指标号）。
- **报告主键与周期**：`<platform>_<org>_<repo>_<workflow>_<period>`，**周期 = 周度**（专家意见 6：触发周期每周；period 形如 `2026W28`）。周度样本量小（Megatron 本例 ~15 run/周），小样本护栏将常态化触发——置信度徽标按周标注；首期基线 202606 为月窗回填特例。workflow version 进 Meta 不进主键。

## 3. 指标体系（v0，全部可由 07-09 管道现有数据计算）

> 锚点与归一化均为**草案**，待基准组（含 cann 侧流水线）回测校准后升定稿。

### 3.1 客观层 CDX_OBJ_*（入分）

| 阶段 | 指标（中文名为权威名） | ID | 口径 | 本期值 |
|---|---|---|---|---|
| C0 | 排队等待时长 | CDX_OBJ_QUEUE_WAIT | 队列/审批占位 job 时长均值（附 max 离群比） | 51s（max 1218s，23.9×） |
| C1 | 绿灯确认时长 | CDX_OBJ_TIME_TO_GREEN | 成功 run 墙钟均值 | 7028s ≈ 117min |
| C1 | 红灯信号时长 | CDX_OBJ_TIME_TO_RED | 失败 run 墙钟均值 | 4561s ≈ 76min |
| C1 | 快速失败占比 | CDX_OBJ_FAST_FAIL_RATE | 失败 run 中 ≤15min 出红灯的占比（专家 2.2：快慢失败体验分档） | 3/13（23.1%） |
| C1 | 空转占比 | CDX_OBJ_IDLE_RATIO | run 内无 job 活跃时间占比（均值） | 37.1% |
| C2 | 门禁前置率 | CDX_OBJ_GATE_COVERAGE | 失败 run 中 root-failure 落在前置检查类 job 的占比 | 3/13（23.1%） |
| C2 | 取消遭遇率 | CDX_OBJ_CANCEL_RATE | 取消 run / 总 run | 16/61（26.2%） |
| C4 | 核心成功率 | CDX_OBJ_SUCCESS_CORE | 成功 /（成功+失败），剔除取消与其他 | 28/41（68.3%） |

> 口径注：本期"快速失败占比"与"门禁前置率"数值相同（被拦=快），但概念不同——前者看失败暴露的**时间**，后者看**阶段位置**；门禁提速或深水区快速失败时二者分离。附录须带此脚注防误读为重复计分。

### 3.2 诊断层 CDX_DIAG_*（log agent 产出；v0 不入分、入证据）

| 指标 | 口径 | 本期值 |
|---|---|---|
| 失败归因分类 | 每失败 run 归入分类法（§3.4） | 已归因 8/13，未归因 5（Phase 1 补齐） |
| 本地复现成本分级 | 失败族可否本地复现（专家 2.1）：L0 纯 CPU 单测 / L1 单卡 / L2 多卡 / L3 仅 CI 级环境 | Callable 族 L0 · FSDP+EP 族 L2 · NCCL 环境族 L3 |
| 根因归因定位率 | 失败族根因定位到 file:line 的覆盖 | 两大失败族均已定位（`argument_utils.py:165`、`megatron/core/utils.py:1084`） |
| 重试税 | 确定性失败被 flaky 重试放大的额外耗时/次 | ~1328s/次（job 82761388031） |
| 漏检代价 | 漏过门禁失败 run 均时 − 拦下均时 | 5906−78 = 5827s/次 |

### 3.3 红旗规则（布尔，触发徽标不入均分）

- **R-FLAG-RETRY**：确定性失败被 flaky 重试 ≥2 轮 → C2 红旗（本期命中）。
- **R-FLAG-QUEUE**：轻量占位 job 排队 max/avg ≥10× → C0 红旗（本期命中，23.9×）。
- 红旗清单开放扩充；每条红旗必须挂诊断层证据。

### 3.4 失败归因分类法（专家 1.4）

失败 run 逐条归入：**代码规范 / 编译失败 / 功能失败（产品代码）/ 工程环境类（环境不满足/环境异常）/ CI 自身稳定性**；不可归因者标"未归因"进管道缺口，禁占位编造。本期实况（N=13）：

| 归因类 | run 数 | 说明 |
|---|---|---|
| 代码规范 | 3（推定） | gate 拦下的 3 个 = linting 拦截（均时 78s）；推定口径 Phase 1 落实 |
| 编译失败 | 0 | container-build 失败率 0% |
| 功能失败（产品代码） | 4 | Callable 类型族 ×3 + FSDP+EP 断言 ×1 |
| 工程环境类 | 1 | NCCL_GRAPH_REGISTER 环境守卫 |
| CI 自身稳定性 | 0 确认 | **失败轨暂未确认；主要嫌疑在取消轨**——16 个取消 run（26.2%）疑似 runner 容量级联，待分型 |
| 未归因 | 5 | Phase 1 逐 run 补齐 |

对专家问题 1.4.2（"哪些是 CI 自身稳定性导致"）的本期答案：**CI 自身稳定性问题主要以"取消"而非"失败"形态出现**，故失败轨与取消轨须并列呈现。

### 3.5 参考量 CDX_REF_*（仅展示）

名义成功率 45.9%（与核心成功率双轨并列）；墙钟 avg 6246s / max 14031s；idle/serial/parallel 三分（37.1%/27.0%/35.9%）。

### 3.6 维护者侧视图（效率 / 稳定性 / 成本，专家 1.1–1.4）

三块面板，全部按 §2.1 工序阶段组织：

1. **工序阶段账单**：每阶段 job 样本数、均时、**合计时长占比**（Σcount×avg 口径）、失败率、判定；行内两层细化展开 job 明细。本期：ST 0.2% / BD 6.3% / UT 45.0% / FT 45.4% / XR 2.2% / ORCH 1.0%——测试类合计约 90% 的 job 时长。
2. **成本视图**（专家 1.3）：每 run 平均 job 数（~37）、每 run job 合计时长（~3.0h，对墙钟 1.7h 即并行度 ~1.8×）、空等构成；**per-PR 成本**（一个 PR 多次 run 的累计消耗）需 PR 关联追踪，列为 Phase 1 缺口不占位。资源单价（GPU/CPU runner 分钟折算）为 v1 增强项。
3. **失败归因分类**（§3.4）+ 取消轨并列。

### 3.7 本期数据缺口（诚实声明，不占位编分）

- **C3 恢复时长**（红灯→同 PR 下一次绿灯）需跨 run 按 PR 关联追踪 → C3 本期不评级，权重重分配并首屏声明。
- per-PR 成本聚合、16 个取消 run 逐条归因、5 个失败 run 归因、job 级 P95 → Phase 1 管道补齐。

## 4. 评分体系（草案）

```
阶段分 = mean(该阶段 CDX_OBJ_* 归一化分)          [红旗不入均分，只挂徽标]
CDX 总分 = 0.10×C0 + 0.30×C1 + 0.30×C2 + 0.10×C3 + 0.20×C4
           [缺口阶段权重按比例重分配给其余阶段，并在首屏声明]
```

- **归一化（草案锚点）**：时长型（分钟）`100 − 35×log10(1 + min/10)`；正向比率 ×100；负向比率 `100 − raw×100`。
- **A–F 等级与星级**：与 SDX/IDX 完全同切点（A≥90 / B 75–89 / C 60–74 / D 40–59 / F<40），描述语换 CI 语境。
- **本期示算（v0.2，收编快速失败占比后）**：C0 98.8(A)🚩 · C1 53.7(D)（v0.1 为 63.9，快速失败 23.1 拉低）· C2 48.5(D)🚩 · C3 缺口 · C4 68.3(C) → **CDX 60.2 / C（压线）**。短板阶段 = C1、C2（两个，触发 v2.1 短板展开规则）。
- **置信度**：run 级 N=61 → 高；失败子集 N=13 → 中；周度出刊后 N/周 ~15，护栏常态化。

## 5. 报告结构（页面分区规范，v0.2 顺序）

1. **Meta 带**：项目 · platform · workflow · 周期（周度） · N · 置信度；回溯统计声明横幅。
2. **诊断摘要**（30 秒决策路径，两类听众共用，置顶）：一句话判决 + CDX 等级卡 + 本期遭遇要点。
3. **KPI 行**（6 卡）。
4. **维护者视图**（先行，专家意见）：工序阶段账单（两层细化）→ 成本视图 → 失败归因分类（失败轨+取消轨并列）→ 流水线依赖图 mini-map（异常徽章，跳转完整诊断工具）。
5. **开发者体验视图**：旅程 C0–C4（导航中枢，短板默认展开）+ 透镜 R 带 + 联动详情（阶段指标 + 痛点 + 证据 + 本地复现成本 + REC 链接）。
6. **改进闭环（REC 卡）**：非空硬门；稳定 ID、优先级、证据等级、两级承接、状态机，下期回填实测值。
7. **趋势与对标**：周度环比（护栏沿 v2.1 规则 7）；对标槽位预置 cann 侧 CI ↔ Megatron-LM CI。
8. **附录**：指标字典（含框架对应列、快慢失败/前置率同值脚注）+ 口径与代际声明。

## 6. 决策物实体（PP / REC）

- 本期从 07-09 RCA 收敛出 **5 条 PP、7 张 REC**（去重后；明细见页面设计稿）。P0 两张：确定性失败关闭 flaky 重试（省 ~1328s/次）、gate 增加运行时冒烟测试（直击门禁前置率 23.1%）。
- **承接团队分型**：REC 天然分流为 CI 维护者（流水线配置/重试策略/门禁设计）与 **产品代码团队**（本期两大失败族都是产品 bug）——体验的根因经常在代码侧。
- PP/REC 跨周期稳定 ID + 状态机接入部署站痛点管理工作流，与 ISSUE 线共用同一套实体机制。

## 7. cann 背景与跨平台适配（先行设计）

后续将出 cann 相关社区（gitcode 侧）CI 体验报告；**cann CI API 预计 2026-07-16 就绪**（专家意见 8）。适配约束：

1. **数据模型平台无关**：指标建立在 run / job / queue / conclusion / duration 抽象上；工序阶段分类法（§2.1）平台无关，逐平台只做 job→阶段映射。GitHub Actions 专属概念（merge queue、matrix leg）标注 **optional 模块**，缺失时标"平台不适用"而非空占位。
2. **指标分级供给**：核心级（run 级 conclusion+时长 → C1/C4 可算）、增强级（job 级遥测 → C0/C2、阶段账单可算）、完整级（日志可得 → CDX_DIAG、失败归因分类、红旗可判）。API 就绪后第一件事 = 三级可得性摸底。
3. **工具链兼容性模块（专家 1.5，场景 profile）**：C++ 仓（cann/runtime、cann/ops-nn）对 gcc 等编译器版本敏感——BD 阶段增设兼容性指标（编译器/工具链版本矩阵覆盖数、版本相关失败占比，归入失败归因"工程环境类"细目）。Megatron 侧同构物是 CUDA/TE/PyTorch 版本敏感（旧代报告已现 TE 版本失配案例），模块设计两侧通用。
4. **对标槽位**：catlass↔cutlass 同构配对延伸到 CI 侧——Megatron-LM 本报告即 NVIDIA 侧基准锚点；横向对标只留槽位不做实体（沿 ISSUE §16.5 边界）。
5. **归一化锚点校准**必须纳入 cann 侧样本后再定稿，避免用 NVIDIA 超算级流水线的时长分布单方面定义"好"。

## 8. 与 07-09 诊断工具的关系

`report/07-09_dep_graph.html` = 诊断层底座（Flow1 log 知识 + Flow2 指标 + Flow3 RCA），**保留并继续演进，不并入体验页**。体验页调取其产物：RCA 判定 → PP/红旗/失败归因；建议池 → REC 收敛；log 知识 → 阶段账单两层细化的"一句话功能"；证据指针 → 下钻链接。管道产出顺序：诊断层先产，体验层派生（生成硬门在体验层校验）。

## 9. 实施路径

- **Phase 0**：口径固化——07-09 管道数据字典成文；工序阶段映射表落库；"未跑/无 metrics" job 采集缺口归因排查（禁生成器合理化叙事）。
- **Phase 1**：管道补齐——PR 关联跨 run 追踪（解锁 C3 + per-PR 成本）、取消/失败 run 逐条归因分型、job 级 P95、失败归因分类自动化；附录 A 框架对应列标注。**并行：cann CI API（~07-16）就绪后做三级数据可得性摸底 + job→工序阶段映射。**
- **Phase 2**：模板与生成规则落地（周度生成硬门、REC 去重收敛、置信度护栏），产出首份周度生产报告。
- **Phase 3**：周期化运营 + cann 侧首报 + 归一化锚点回测定稿 + 对标槽位激活。
