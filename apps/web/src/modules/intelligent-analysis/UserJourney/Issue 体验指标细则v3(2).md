# Issue 体验指标细则v3

## 1. 评价目标

本体系用于评价：

> 外部开发者提交 Issue 后，是否获得及时响应、有效沟通、持续推进、合理结果和适当闭环。

评分对象分为两个层级：

- **单条 Issue**：计算该 Issue 的体验分、维度分和风险标签。
- **单个仓库周度结果**：聚合本周符合条件的 Issue，形成仓库体验分和问题分析。

本阶段不评价 PR 提交、Review、CI 或合并体验。

与 Issue 解决直接相关的 PR，可以作为以下事项的客观证据：

- 已采取实质处理动作；
- 已形成修复结果；
- 修复已经合并；
- Issue 关闭具备合理依据。

评分必须按照以下顺序执行：

```text
样本分类
→ 分配分析池
→ 判断指标是否适用
→ 计算客观事实
→ 识别语义事件
→ 进行必要的质量判断
→ 计算单项指标分
→ 计算单条 Issue 分
→ 应用严重问题控制
→ 聚合仓库周度结果
```

------

## 2. 核心评分原则

### 2.1 只评价外部开发者体验

仓库规模、维护者数量、Issue 突增等信息可以作为背景解释，但不能直接为体验结果加分。

从用户视角看，社区资源不足可以解释响应慢，但不会消除响应慢带来的体验影响。

### 2.2 客观数据优先

能够通过时间、评论、状态和事件直接计算的指标，不交给 LLM 自由评分。

### 2.3 LLM 负责识别事实，而不是决定总分

LLM 主要完成：

- Issue 样本分类；
- 用户诉求识别；
- 实质回应识别；
- 信息请求识别；
- 用户催促识别；
- 结果形成时间识别；
- 关闭原因识别；
- 证据与用户目标的关联判断。

最终分数必须由固定算法计算。

### 2.4 不适用不等于零分

某项指标不适用于当前 Issue 时：

```text
not_applicable = true
```

该指标：

- 不计零分；
- 不进入分母；
- 不影响总分。

### 2.5 缺失不等于满分

无法获取数据或模型无法可靠判断时，不能默认给满分。

应标记为：

- 数据缺失；
- 低置信度；
- 待人工复核；
- 暂不参与正式评分。

------

## 3. 第一步：判断 Issue 是否进入评分

每条 Issue 必须先完成样本分类。

### 3.1 样本分类标准

| 样本类型          | 定义                                                         | 是否进入主评分 |
| ----------------- | ------------------------------------------------------------ | -------------- |
| 外部开发者 Issue  | 外部开发者围绕项目使用、部署、运行、文档、兼容性或产品能力提出真实诉求 | 是             |
| 内部开发 Issue    | 项目成员或内部协作者创建的研发、测试、版本规划、重构或技术改造任务 | 否             |
| 社区流程 Issue    | 用于登记、审批、同步、占位、关联或管理流程，本身不承载独立用户诉求 | 否             |
| 自动化 Issue      | 由已确认的 Bot、扫描程序、流水线或自动化系统创建             | 否             |
| Spam 或无效 Issue | 广告、推广、恶意灌水、测试垃圾、无关内容或无法形成有效诉求   | 否             |
| 无法判断          | 作者身份、内容或创建意图证据不足                             | 暂不评分       |

### 3.2 外部开发者 Issue 的判断条件

一条 Issue 通常应同时满足：

1. 提交者不是已确认的仓库成员、内部协作者或 Bot；
2. 正文表达了与项目相关的真实问题或诉求；
3. 即使去除关联 PR、审批或管理流程，该诉求仍然独立成立；
4. 该 Issue 需要社区提供回答、判断、处理或结果。

外部开发者 Issue 可以包括：

- Bug 或异常；
- 安装、编译和部署问题；
- 环境与依赖问题；
- API 或功能使用咨询；
- 文档问题；
- 性能或兼容性问题；
- 功能建议；
- 其他有效外部反馈。

### 3.3 内部开发与社区流程的区分

使用以下问题判断：

> 去掉与该 Issue 关联的审批、PR、登记或管理对象后，这条 Issue 是否仍代表一项需要独立完成的工程任务？

- **仍然成立**：内部开发 Issue；
- **不再成立**：社区流程 Issue。

### 3.4 分类置信度

| 分类置信度 | 处理方式                     |
| ---------- | ---------------------------- |
| ≥0.80      | 正式采用分类结果             |
| 0.60～0.79 | 暂时采用，同时进入人工抽检池 |
| <0.60      | 不进入正式评分，转人工复核   |

低置信度不是一种样本类型。

模型不得因为必须输出结果而强制选择最接近的类别。

------

## 4. 第二步：分配分析池

只有“外部开发者 Issue”进入体验评分。

同一条 Issue 可以进入多个分析池，但每个池的分析目的不同。

| 分析池 | 进入条件                                         | 主要评价内容                 |
| ------ | ------------------------------------------------ | ---------------------------- |
| 新增池 | Issue 在本周期内创建                             | 初始响应和首次实质回应       |
| 过程池 | 本周期发生有效评论、处理动作或状态变化           | 沟通、等待、推进、承诺和交接 |
| 结果池 | 本周期形成明确结果、关闭、重开，或达到结果成熟期 | 结果、验证、稳定性和关闭     |

### 4.1 新增池

进入条件：

```text
issue_created_at ∈ 当前统计周期
```

主要使用的指标：

- 首次社区侧响应时效；
- 首次实质回应时效；
- 用户目标理解；
- 初次信息请求效率。

### 4.2 过程池

进入条件为本周期内至少发生一项有效处理事件，例如：

- Maintainer 发表有效评论；
- 用户针对处理结果进行回复；
- 添加或变更 Assignee；
- 发生标签、关闭或重开操作；
- 关联与解决相关的 PR；
- 形成新的处理结论；
- 出现明确承诺或进度更新。

纯自动通知不单独构成有效互动。

### 4.3 结果池

满足以下任一条件即可进入：

- 本周期首次形成明确结果；
- 本周期关闭；
- 本周期重开；
- 创建已满 30 天，但仍未形成明确结果；
- 用户明确表示已有结果无效；
- 结果形成后出现稳定性问题。

### 4.4 分析池去重规则

同一条 Issue 可以同时进入新增池、过程池和结果池。

但是：

> 同一条 Issue 在同一个指标中只能贡献一次。

禁止采用以下算法：

```text
仓库总分
=（新增池平均分 + 过程池平均分 + 结果池平均分）÷ 3
```

正确方式是：

> 每个指标在自己的有效 Issue 集合中独立聚合，再按照指标权重计算仓库总分。

------

## 5. 第三步：识别可观察里程碑

Issue 不需要严格按照固定生命周期发展。

系统只识别已经出现的可观察里程碑。

| 里程碑          | 定义                                                 |
| --------------- | ---------------------------------------------------- |
| M1 已提交       | Issue 已创建                                         |
| M2 社区侧响应   | 出现可归属于社区处理侧的评论或操作                   |
| M3 实质回应     | 出现判断、必要信息请求、排查方案、解决方案或明确处置 |
| M4 处理路径明确 | 下一步动作、等待条件或责任方向已经明确               |
| M5 形成结果     | 已给出答案、方案、结论、边界说明或有效转交           |
| M6 获得验证     | 用户确认、客观证据验证，或出现明确未解决反馈         |
| M7 已关闭       | Issue 状态变为关闭                                   |
| M8 已重开       | Issue 关闭后重新打开                                 |

里程碑规则：

- 不要求全部发生；
- 不要求严格按照顺序发生；
- 同一条评论可以触发多个里程碑；
- 里程碑只表示事实发生，不直接代表体验良好。

------

## 6. 第四步：判断指标是否适用

每项指标独立判断适用条件。

| 指标               | 适用条件                                   |
| ------------------ | ------------------------------------------ |
| 首次社区侧响应时效 | 已发生社区侧响应，或 Issue 创建满 24 小时  |
| 首次实质回应时效   | 已发生实质回应，或 Issue 创建满 72 小时    |
| 用户目标理解       | 已出现社区侧实质回复                       |
| 信息请求效率       | 社区实际请求用户补充信息                   |
| 沟通表达清晰度     | 已出现包含解释、判断或操作建议的回复       |
| 可避免用户跟进     | 存在用户后续回复候选                       |
| 处理路径明确性     | 已发生实质回应，或已进入持续处理过程       |
| 最长无说明停滞     | 尚未形成结果，且存在可观察处理过程         |
| 承诺兑现度         | 社区作出明确行动或时间承诺                 |
| 实质处理动作及时性 | 已发生实质处理动作，或创建满 30 天仍无动作 |
| 交接连续性         | 负责人、团队或处理方发生变化               |
| 用户目标处理结果   | 已形成结果，或创建满 30 天仍无结果         |
| 结果证据强度       | 已形成结果                                 |
| 形成结果时效       | 已形成结果，或达到 30 天成熟期             |
| 结果验证强度       | 已形成结果且存在合理验证机会               |
| 结果稳定性         | 已关闭、重开或出现结果失效信号             |
| 关闭原因明确性     | Issue 已关闭                               |
| 关闭时机合理性     | Issue 已关闭                               |
| 后续路径完整性     | Issue 已关闭，且仍可能需要继续反馈         |

------

## 7. 第五步：按三层证据架构评分

### 7.1 第一层：客观事实

由程序直接计算，不交给 LLM 自由判断。

包括：

- Issue 创建时间；
- 评论时间；
- 更新时间；
- 关闭和重开时间；
- 评论顺序；
- 评论者身份；
- 标签变化；
- Assignee 变化；
- 状态变化；
- 操作日志；
- PR 关联和合并状态；
- 最长无事件间隔；
- 已确认的平台 Bot 身份。

### 7.2 第二层：语义事件

LLM 负责将非结构化内容识别为结构化事件。

包括：

- 首次实质回应是哪条评论；
- 社区是否请求补充信息；
- 请求的信息是否已经提供；
- 是否出现明确承诺；
- 用户是否在催促或追问进度；
- 用户是否确认问题解决；
- 用户是否明确表示未解决；
- 何时首次形成明确结果；
- 关闭原因是什么；
- 关联 PR 是否与当前 Issue 的用户目标直接相关。

识别完成后，时间差和次数仍由程序计算。

### 7.3 第三层：体验质量判断

仅对无法客观化的内容使用 LLM 评分。

统一采用六档分值：

| 分数 | 统一含义                                       |
| ---- | ---------------------------------------------- |
| 100  | 完整实现体验目标，没有明显缺陷                 |
| 80   | 整体良好，仅存在轻微问题                       |
| 60   | 基本合格，但存在明显体验成本                   |
| 40   | 只完成部分目标，主要缺口仍然存在               |
| 20   | 基本失败，但存在少量相关行为                   |
| 0    | 达到评价条件后行为完全缺失，或存在明确相反事实 |

0 分应优先由确定性规则触发，不应成为 LLM 常规使用的主观档位。

------

## 8. 评分指标与权重

| 一级维度          | 权重     |
| ----------------- | -------- |
| A. 响应可达性     | 20%      |
| B. 理解与沟通     | 20%      |
| C. 推进与透明度   | 20%      |
| D. 结果与解决质量 | 30%      |
| E. 关闭与后续体验 | 10%      |
| **合计**          | **100%** |

### 8.1 A. 响应可达性：20%

#### A1. 首次社区侧响应时效：8%

##### 定义

从 Issue 创建时间到首次可高置信度归属于社区处理侧的响应时间。

以下内容不计入：

- Issue 作者自己补充信息；
- 普通外部用户表示“我也遇到了”；
- 与问题无关的评论；
- 纯系统状态通知；
- 无法确认属于社区处理侧的账号回复。

##### 评分标准

| 响应时间               | 得分 |
| ---------------------- | ---- |
| ≤24 小时               | 100  |
| 24～48 小时            | 80   |
| 48～72 小时            | 60   |
| 3～7 天                | 40   |
| 7～14 天               | 20   |
| >14 天或成熟后仍无响应 | 0    |

#### A2. 首次实质回应时效：12%

##### 定义

从 Issue 创建时间到首次能够实际推动处理的社区回应时间。

实质回应至少包含以下一项：

- 对问题作出初步判断；
- 明确请求必要信息；
- 给出排查方法；
- 提供可执行方案；
- 给出明确结论；
- 明确处理方向或下一步。

以下内容不属于实质回应：

- “收到”；
- “感谢反馈”；
- “我们看看”；
- 单纯添加标签；
- 无具体内容的模板回复；
- 没有说明具体需要什么的“请补充信息”。

##### 评分标准

| 实质回应时间       | 得分 |
| ------------------ | ---- |
| ≤48 小时           | 100  |
| 2～4 天            | 80   |
| 4～7 天            | 60   |
| 7～14 天           | 40   |
| 14～30 天          | 20   |
| 成熟后仍无实质回应 | 0    |

### 8.2 B. 理解与沟通：20%

#### B1. 用户主要目标理解准确性：6%

评价社区是否正确理解用户最希望解决的问题。

判断内容包括：

- 是否识别主要问题；
- 是否识别用户期望的结果；
- 是否遗漏关键环境或限制条件；
- 是否因理解错误造成额外沟通；
- 是否回答了次要问题，却忽略主要阻塞。

按统一六档标准评分。

#### B2. 信息请求效率：5%

仅在社区请求用户补充信息时参与评分。

##### 基础评分

| 有效信息请求轮次 | 基础得分 |
| ---------------- | -------- |
| 1 轮             | 100      |
| 2 轮             | 80       |
| 3 轮             | 60       |
| 4 轮             | 40       |
| ≥5 轮            | 20       |

##### 调整规则

出现以下情况时，下调一个或多个档位：

- 重复请求正文中已经提供的信息；
- 请求内容过于宽泛；
- 没有说明需要什么；
- 请求与问题定位无明显关系；
- 多个处理者重复索取相同信息。

以下情况不应扣分：

- 用户首次提供的信息确实不足；
- 后续技术发现产生新的合理信息需求；
- 问题复杂度客观要求分阶段收集信息。

多次重复索取已有信息，并导致 Issue 无法推进时，可以记 0 分。

#### B3. 沟通表达清晰度：5%

评价社区回复是否：

- 容易理解；
- 结论明确；
- 步骤清晰；
- 命令、参数、版本或配置准确；
- 必要术语有解释；
- 不存在明显歧义；
- 不存在前后矛盾。

按统一六档标准评分。

#### B4. 可避免用户跟进负担：4%

程序先生成用户后续评论候选，LLM 判断哪些属于可避免跟进。

可避免跟进包括：

- 询问是否有人处理；
- 询问当前进度；
- 因社区长期无回应而重复描述问题；
- 因上下文丢失而重新提供相同信息；
- 因回复含糊而反复确认下一步。

以下内容不计入：

- 用户主动补充新的技术发现；
- 按要求提供日志；
- 验证解决方案；
- 正常技术讨论；
- 对新的社区问题进行补充。

##### 评分标准

| 可避免跟进次数       | 得分 |
| -------------------- | ---- |
| 0 次                 | 100  |
| 1 次                 | 80   |
| 2 次                 | 60   |
| 3 次                 | 40   |
| ≥4 次                | 20   |
| 持续失联导致无法推进 | 0    |

### 8.3 C. 推进与透明度：20%

#### C1. 处理路径明确性：4%

评价用户是否能够知道：

- 下一步将执行什么动作；
- 当前在等待什么；
- 由谁继续处理；
- 用户是否需要继续操作；
- 什么条件下会获得下一次更新。

按统一六档标准评分。

只有“正在看”“后续处理”等模糊表达时，一般不高于 40 分。

#### C2. 最长无说明停滞时长：7%

##### 定义

在尚未形成结果期间，计算两个有效处理事件之间最长的无合理说明停滞时间。

有效处理事件包括：

- 实质评论；
- 明确进度更新；
- 状态或责任变化；
- 解决相关 PR 事件；
- 明确处理结论；
- 用户按要求补充必要信息。

##### 暂停计时条件

以下情况可以暂停停滞计时：

- 已明确等待用户回复；
- 已明确等待版本发布；
- 已明确等待其他团队或外部条件；
- 已说明延期原因；
- 已提供下一次评估时间或触发条件。

“等待用户回复”仅在请求内容明确且必要时成立。

##### 评分标准

| 最长无说明停滞 | 得分 |
| -------------- | ---- |
| ≤3 天          | 100  |
| 3～7 天        | 80   |
| 7～14 天       | 60   |
| 14～30 天      | 40   |
| 30～60 天      | 20   |
| >60 天         | 0    |

#### C3. 承诺兑现度：4%

仅在社区作出明确承诺时参与评分。

明确承诺包括：

- 明确完成日期；
- 明确回复时间；
- 明确版本节点；
- 明确将执行某项动作；
- 明确转交给某个处理方。

不包含：

- “有空看看”；
- “后续考虑”；
- “可能会处理”；
- 没有行动对象的模糊意向。

##### 评分标准

| 承诺结果                 | 得分 |
| ------------------------ | ---- |
| 按期或提前完成           | 100  |
| 轻微延期，并主动说明     | 80   |
| 明显延期，但持续提供更新 | 60   |
| 逾期后需要用户跟进       | 40   |
| 承诺后基本失联           | 20   |
| 长期未兑现且无解释       | 0    |

#### C4. 实质处理动作及时性：3%

##### 定义

从 Issue 创建到首次发生实质处理动作的时间。

实质处理动作包括：

- 明确诊断；
- 给出解决方向；
- 提供可执行方案；
- 作出正式处置决定；
- 明确有效转交；
- 关联与当前用户目标直接相关的解决 PR。

##### 评分标准

| 首次实质处理动作时间     | 得分 |
| ------------------------ | ---- |
| ≤3 天                    | 100  |
| 3～7 天                  | 80   |
| 7～14 天                 | 60   |
| 14～30 天                | 40   |
| 30～60 天                | 20   |
| 成熟后无任何实质处理动作 | 0    |

#### C5. 交接连续性：2%

仅在 Assignee、团队或实际处理方发生变化时参与评分。

评价内容包括：

- 新承接方是否明确；
- 原有上下文是否保留；
- 用户是否需要重新解释；
- 交接后是否继续推进；
- 是否出现无人负责。

按统一六档标准评分。

### 8.4 D. 结果与解决质量：30%

#### D1. 用户目标处理结果：10%

评价用户主要目标最终是否得到有效满足，或在无法满足时获得合理处置。

高分不要求所有 Issue 都完成代码修复。

以下结果均可能获得高分：

- 问题被修复；
- 提供有效规避方案；
- 给出正确且可执行的答案；
- 明确项目能力边界；
- 合理拒绝功能建议；
- 明确延期及原因；
- 转交到真实负责方；
- 无法复现，但说明充分并提供重新开启条件；
- 重复 Issue 被正确关联到已有解决路径。

评分重点包括：

- 是否回应用户主要目标；
- 是否真正解决或合理处置；
- 用户是否仍被阻塞；
- 结果是否准确；
- 是否存在明显未解决信号。

达到结果成熟期后仍完全没有结果，记 0 分。

#### D2. 结果证据强度：6%

结果证据按强度分类。

##### 强证据

- 与用户目标直接相关的修复 PR 已合并；
- 有明确 Patch、Commit 或修复版本；
- 用户明确确认问题解决；
- 提供可复现的测试结果；
- Maintainer 给出完整验证记录。

##### 中等证据

- 有相关 PR，但尚未合并；
- 提供具体配置、命令或规避方案；
- 提供有效文档；
- 提供明确版本迁移路径；
- 给出具体的复现或诊断结果。

##### 弱证据

- 只有口头结论；
- 仅表示“已经解决”；
- 没有给出具体依据；
- 结果无法独立验证。

##### 评分标准

| 证据强度           | 得分 |
| ------------------ | ---- |
| 多个强证据相互支持 | 100  |
| 至少一个强证据     | 80   |
| 有明确中等证据     | 60   |
| 只有较弱说明       | 40   |
| 几乎没有依据       | 20   |
| 证据与结论明显冲突 | 0    |

关联 PR 不直接加分。

只有在确认该 PR 与当前 Issue 的用户目标直接相关时，才能作为结果证据。

#### D3. 形成结果时效：6%

##### 定义

从 Issue 创建到首次形成明确结果的时间。

明确结果包括：

- 给出有效答案；
- 提供解决方案；
- 提供规避方案；
- 确认修复；
- 明确项目边界；
- 合理拒绝；
- 明确延期；
- 有效转交；
- 给出无法复现结论及继续处理条件。

“正在处理”“后续考虑”不属于明确结果。

##### 评分标准

| 形成结果时间       | 得分 |
| ------------------ | ---- |
| ≤3 天              | 100  |
| 3～7 天            | 80   |
| 7～14 天           | 60   |
| 14～30 天          | 40   |
| 30～60 天          | 20   |
| 成熟后仍无明确结果 | 0    |

#### D4. 结果验证强度：4%

##### 评分标准

| 验证情况                     | 得分 |
| ---------------------------- | ---- |
| 用户确认，且有客观证据       | 100  |
| 有较强客观验证证据           | 80   |
| 已形成结果，但缺少进一步验证 | 60   |
| 存在部分未解决信号           | 40   |
| 用户明确表示结果无效         | 20   |
| 已知结果无效，仍作为成功处理 | 0    |

用户没有继续回复，不得自动认定为问题已经解决。

#### D5. 结果稳定性：4%

主要使用以下事实：

- 是否因同一问题重开；
- 是否出现明确未解决反馈；
- 是否多次关闭和重开；
- 修复后是否很快出现失败信号；
- 用户是否在关闭后继续报告相同问题。

##### 评分标准

| 情况                     | 得分 |
| ------------------------ | ---- |
| 无结果失效或异常重开信号 | 100  |
| 正常流程性重开           | 80   |
| 因结果不完整重开一次     | 60   |
| 因明显未解决或误关闭重开 | 40   |
| 多次因结果无效重开       | 20   |
| 多次错误关闭与重开       | 0    |

正常流程性重开不应视为体验失败。

### 8.5 E. 关闭与后续体验：10%

#### E1. 关闭原因明确性：3%

评价用户是否能够明确知道 Issue 为什么关闭。

关闭原因应尽量归入以下类别：

- 已解决；
- 已回答；
- 重复问题；
- 不支持或不采纳；
- 无法复现；
- 等待用户信息超时；
- 转移至其他仓库或渠道；
- 长期无活动关闭；
- 误关闭；
- 无解释关闭；
- 原因不明。

按统一六档标准评分。

完全无解释关闭记 0 分。

#### E2. 关闭时机合理性：4%

综合判断：

- 是否已经形成明确结果；
- 是否具备足够证据；
- 用户是否明确表示未解决；
- 是否已合理等待用户回复；
- 是否存在与解决相关的 PR 合并；
- 关闭后是否很快因未解决重开；
- 是否在用户刚补充必要信息后直接关闭。

按统一六档标准评分。

明知问题未解决，仍宣称已经解决并关闭，应记 0 分，并触发严重问题控制。

#### E3. 后续路径完整性：3%

仅在关闭后仍可能需要继续反馈时参与评分。

评价是否明确说明：

- 重新开启条件；
- 正确的反馈入口；
- 其他负责仓库或团队；
- 后续版本；
- 重新评估条件；
- 需要用户补充的材料。

问题已经完整解决并得到用户确认时，可以标记为不适用。

## 9. 单条 Issue 分数计算

### 9.1 指标分

每个适用指标得到 0～100 分。

### 9.2 原始体验分

```text
raw_score
= Σ（指标得分 × 指标权重）
÷ Σ（适用指标权重）
```

### 9.3 评分覆盖率

```text
coverage
= Σ（适用指标权重）
÷ 100
```

例如：

```json
{
  "raw_score": 82,
  "coverage": 0.72
}
```

结果应表述为：

> 阶段性体验分 82/100，当前评分覆盖率 72%。

评分覆盖率达到 80% 后，才建议称为“完整体验分”。

### 9.4 维度分

每个一级维度单独计算：

```text
维度分
= Σ（该维度适用指标得分 × 指标权重）
÷ Σ（该维度适用指标权重）
```

维度完全没有适用指标时，不输出该维度分。

## 10. 严重问题控制

严重问题控制用于防止核心体验失败被其他高分抵消。

计算方式：

```text
final_score = min(raw_score, score_cap)
```

严重问题只限制最高分，不再重复扣固定分。

### 10.1 一级严重问题

最高分限制为 39 分。

典型情况：

- 攻击、羞辱或歧视用户；
- 恶意误导；
- 明知结果无效仍宣称问题已解决；
- 篡改或歪曲用户诉求；
- 多次错误关闭，并拒绝提供继续反馈路径。

### 10.2 二级严重问题

最高分限制为 59 分。

典型情况：

- 超过 14 天没有任何社区侧响应；
- 完全无解释关闭；
- 用户明确表示未解决后仍关闭；
- 转交后彻底失去责任方；
- 多次重复索取已经提供的信息；
- 自动关闭后没有人工介入或重新开启路径。

### 10.3 三级重大体验风险

最高分限制为 79 分。

典型情况：

- 超过 30 天没有实质推进；
- 多次可避免的用户催促；
- 严重承诺违约；
- 多次交接并丢失上下文；
- 处理路径长期不清晰；
- Issue 长期停滞且没有说明。

### 10.4 严重问题采用条件

严重问题必须同时满足：

- 有明确评论或事件证据；
- 触发原因清晰；
- 判断置信度达到正式采用标准；
- 进入人工抽检范围。

## 11. 单条 Issue 体验等级

| 最终分  | 等级     |
| ------- | -------- |
| 90～100 | 优秀     |
| 80～89  | 良好     |
| 60～79  | 待改进   |
| 0～59   | 严重问题 |

评分覆盖率不足 80% 时，应增加“阶段性”前缀。

例如：

> 阶段性良好，体验分 84，评分覆盖率 65%。

严重问题控制触发后，等级按照封顶后的最终分计算。

## 12. 仓库周度分数计算

### 12.1 指标独立聚合

每个指标只使用满足该指标适用条件的 Issue。

```text
仓库指标分
= Σ 单条 Issue 指标分
÷ 该指标成功评分的 Issue 数
```

每个仓库指标必须同时记录：

- `eligible_n`：满足指标适用条件的 Issue 数；
- `scored_n`：成功获得正式分数的 Issue 数；
- `not_applicable_n`：不适用的 Issue 数；
- `low_confidence_n`：低置信度结果数；
- `low_score_n`：低于 60 分的 Issue 数；
- `severe_issue_n`：触发严重问题控制的 Issue 数。

建议同时展示：

- 平均分；
- 中位数；
- 低分 Issue 占比；
- 严重问题率。

### 12.2 仓库总分

```text
repository_score
= Σ（仓库指标分 × 指标权重）
÷ Σ（本周期存在有效样本的指标权重）
```

### 12.3 仓库评分覆盖率

```text
repository_coverage
= Σ（本周期存在有效样本的指标权重）
÷ 100
```

禁止：

- 用全部 Issue 数作为所有指标的统一分母；
- 将不适用指标计为 0 分；
- 将缺失指标补为 100 分；
- 将新增池、过程池和结果池简单平均；
- 仅展示仓库总分而隐藏指标样本量。

## 13. 仓库周度等级与少样本规则

按本周有效外部开发者 Issue 数决定发布方式。

| 本周有效样本数 | 发布方式                                   |
| -------------- | ------------------------------------------ |
| ≥10            | 发布正式周度分数和等级                     |
| 5～9           | 发布分数，标记“低样本，谨慎比较”           |
| 1～4           | 不发布正式周度等级，展示个案和滚动四周结果 |
| 0              | 不发布体验分，标记“本周无有效外部体验样本” |

建议同步计算滚动四周分数。

滚动四周有效样本仍少于 5 条时：

- 不发布正式等级；
- 展示指标观察值；
- 展示代表性 Issue；
- 展示严重问题；
- 明确说明样本不足。

少样本不能通过混入内部开发 Issue 或流程 Issue 来增加数量。

## 14. LLM 判断结果要求

每个语义判断至少输出：

```json
{
  "value": "结构化枚举或事件类型",
  "confidence": 0.92,
  "not_applicable": false,
  "evidence": [
    {
      "source_type": "comment",
      "source_id": "comment_123",
      "reason": "该评论首次提供了明确的排查步骤"
    }
  ],
  "reason": "简洁、可审计的判断说明"
}
```

置信度处理规则：

| 置信度     | 处理方式                   |
| ---------- | -------------------------- |
| ≥0.80      | 正常计分                   |
| 0.60～0.79 | 暂时计分，同时进入人工抽检 |
| <0.60      | 不进入正式分数，标记待复核 |

所有关键时间锚点必须保留对应证据 ID，例如：

- 首次实质回应评论 ID；
- 首次结果评论 ID；
- 用户确认评论 ID；
- 未解决反馈评论 ID；
- 关闭事件 ID；
- 重开事件 ID；
- 相关 PR ID。

## 15. 执行结论

完整评分流程必须遵循：

```text
判断 Issue 是否代表外部开发者体验
→ 分配新增池、过程池和结果池
→ 判断每项指标是否适用
→ 优先计算客观事实
→ 使用 LLM 识别语义事件
→ 仅对无法客观化的质量进行分档判断
→ 按固定规则计算单项分数
→ 计算单条 Issue 原始分和覆盖率
→ 应用严重问题封顶
→ 按指标有效样本聚合仓库周度结果
→ 结合样本量、覆盖率和严重问题解释结果
```

核心规则可以概括为：

> 能由数据直接计算的，不让 LLM 打分；能由 LLM 识别为事实的，不让 LLM 自由裁量；只有真正无法客观化的体验质量，才使用统一分档评价。

这套指南作为 V1 主评分口径。

以下内容应单独维护为工程实现规范，不与主评分指南混在一起：

- Prompt；
- JSON Schema；
- 社区侧账号识别规则；
- 语义事件判定样例；
- 数据异常处理；
- 模型版本；
- 指标版本；
- 阈值版本；
- 人工抽检流程。
