# Issue 体验指标体系 V2+（最终版）

> 5 阶段 19 项指标，每项指标评价一个独立维度，指标间解耦。

## 1. 概述

### 1.1 核心设计

- **5 个阶段**：创建 → 分配与首次响应 → 讨论与解决 → 总结与关闭 → 自动化治理
- **客观指标**：由程序基于规则和时间差直接计算
- **混合客观指标**：LLM 识别事实 + 程序计算分数
- **主观指标**：由 LLM 进行六档分（100/80/60/40/20/0）主观质量判断
- **不适用处理**：指标不适用时不计零分、不进分母、不影响总分
- **痛点阈值**：阶段分低于 60 分标记为痛点

### 1.2 阶段总览

| 阶段 | 名称 | 客观 | 主观 | 合计 | 客观占比 | 主观占比 |
|------|------|------|------|------|---------|---------|
| 1 | Creation 创建 | 0 | 2 | 2 | 0% | 100% |
| 2 | Triage & Response 分配与首次响应 | 2 | 2 | 4 | 60% | 40% |
| 3 | Discussion & Resolution 讨论与解决 | 2 | 2 | 4 | 60% | 40% |
| 4 | Closure 总结与关闭 | 2 | 2 | 4 | 60% | 40% |
| 5 | Bot/Agent Governance 自动化治理 | 2 | 3 | 5 | 60% | 40% |
| **合计** | | **8** | **11** | **19** | | |

### 1.3 阶段权重（IDX 总分计算）

| 阶段 | IDX 权重 |
|------|---------|
| 1. 创建 | 15% |
| 2. 分配与首次响应 | 20% |
| 3. 讨论与解决 | 25% |
| 4. 总结与关闭 | 20% |
| 5. 自动化治理 | 0%（参考项） |
| **合计** | **100%** |

### 1.4 时效指标链条

```
创建 → 响应(≤2h) → 实质回应(≤48h) → 形成结果(≤3天) → 关闭
       阶段2          阶段2             阶段3           阶段4
```

### 1.5 三种计算类型

| 类型 | 数量 | 计算方式 | 涉及指标 |
|------|------|---------|---------|
| **纯客观** | 5 项 | 程序按规则累加/时间差分档 | `obj-response-speed`、`obj-solution-evidence`、`obj-decision-transparency`、`obj-closure-reuse`、`obj-bot-governance` |
| **混合客观** | 2 项 | LLM识别语义事件 + 程序算时间差分档 | `obj-first-substantive-response-time`、`obj-result-formation-timeliness` |
| **纯主观** | 12 项 | LLM 六档评分（100/80/60/40/20/0） | 所有 `sub-*` 指标 |

> 合计 8 个客观指标（5 纯客观 + 2 混合客观 + 1 Bot反向分）+ 11 个主观指标 = 19 项。

### 1.6 LLM 六档分值标准

| 分数 | 含义 |
|------|------|
| 100 | 完整实现体验目标，没有明显缺陷 |
| 80 | 整体良好，仅存在轻微问题 |
| 60 | 基本合格，但存在明显体验成本 |
| 40 | 只完成部分目标，主要缺口仍然存在 |
| 20 | 基本失败，但存在少量相关行为 |
| 0 | 达到评价条件后行为完全缺失，或存在明确相反事实 |

## 2. 阶段 1：创建（Creation）

评价 Issue 本身（用户提交的内容）。

- **起点**：Issue 被创建
- **终点**：社区开始首次响应（评论/打标签/分配 Assignee）
- **核心问题**：这条 Issue 是否具备被有效处理的前提条件？

**混合比例**：主观 100%

### 指标

#### 1. `sub-input-quality` 输入质量

**指标含义**：衡量用户提交的 Issue 内容是否完整、信息是否充分，是否具备被开发者有效处理的前提条件。检查是否包含复现步骤、环境信息、日志代码、预期与实际对比、结构化章节等关键要素。

| 分数 | 判定条件 |
|------|---------|
| 100 | 完整包含复现步骤+环境信息+日志代码+预期对比+结构化章节，5项全有 |
| 80 | 含4项，仅缺1项次要信息（如缺日志但有复现步骤+环境+预期对比+结构化） |
| 60 | 含3项核心信息（复现步骤+环境+预期对比），但缺日志代码或结构化章节 |
| 40 | 含2项（如仅有标题+简短描述），缺少复现步骤和环境信息 |
| 20 | 仅有1句模糊描述，无法据内容判断问题 |
| 0 | 正文为空或完全无关内容 |

#### 2. `sub-agent-noise-risk` AI噪音风险

**指标含义**：衡量 Issue 内容是否为真实用户诉求，还是 AI 工具生成的低质量/幻觉内容。高分表示内容真实有效，低分表示疑似 AI 生成的虚假或无意义内容。

| 分数 | 判定条件 |
|------|---------|
| 100 | 内容真实具体，含明确的个人使用场景/版本号/报错信息，无AI幻觉迹象 |
| 80 | 内容真实，但表述偏模板化；含具体技术细节，无明显幻觉 |
| 60 | 内容部分真实，但存在AI辅助生成痕迹（格式过于工整、措辞通用化），无幻觉 |
| 40 | 疑似AI生成，内容通用化，缺少具体场景，但技术方向大致正确 |
| 20 | 明显AI生成，含幻觉信息（引用不存在的API/版本/功能），但部分内容有参考价值 |
| 0 | 纯AI生成噪音，内容虚假、无实际意义、与仓库无关 |

**不适用条件**：无（所有 Issue 都有内容）

## 3. 阶段 2：分配与首次响应（Triage & Response）

评价社区接住 Issue 的过程。

- **起点**：社区首次介入（评论/打标签/分配 Assignee）
- **终点**：进入实质讨论或开始动手处理
- **核心问题**：社区有没有及时且有效地"接住"这个 Issue？

**混合比例**：客观 60% + 主观 40%

### 指标

#### 3. `obj-response-speed` 响应速度

**指标含义**：衡量社区在 Issue 创建后多久给出首次回复，反映"有没有人管"。只看首条非作者评论距创建的时间差，不评估回复内容质量。

| 分数 | 判定条件 |
|------|---------|
| 100 | 首条非作者评论 ≤2h |
| 80 | ≤24h |
| 60 | ≤72h |
| 40 | ≤7d |
| 20 | ≤14d |
| 0 | >14d 或无评论 |

#### 4. `obj-first-substantive-response-time` 首次实质回应时效

**指标含义**：衡量社区多久给出"有实质内容"的回复（如排查方向、技术分析、方案建议），而非简单的"收到"或"+1"。由 LLM 识别哪条评论是实质回应，再由程序计算时间差。

| 分数 | 判定条件 |
|------|---------|
| 100 | 首次实质回应 ≤48h |
| 80 | ≤4d |
| 60 | ≤7d |
| 40 | ≤14d |
| 20 | ≤30d |
| 0 | >30d 或无实质回应 |

#### 5. `sub-triage-routing-correctness` 分流正确性

**指标含义**：衡量 Issue 是否被正确打标签、归类和重定向，是否进入了正确的处理路径（如 bug 进 bug 流程、需求进需求流程）。评价分流"方向对不对"，不评价后续处理快慢。

| 分数 | 判定条件 |
|------|---------|
| 100 | 标签+归类+重定向全正确，Issue 进入最佳处理路径且有明确类型标签 |
| 80 | 标签和归类正确，但缺少1个辅助标签（如缺优先级标签但有类型标签） |
| 60 | 有基本类型标签，但归类不够精确（如 bug 标为 question 但方向正确） |
| 40 | 有标签但标签错误或误导（如需求标为 bug），需人工纠正 |
| 20 | 无标签无归类，但有 assignee 指派，路径模糊 |
| 0 | 无标签、无归类、无指派、无重定向，Issue 完全未进入任何处理路径 |

#### 6. `sub-triage-owner-clarity` 责任归属清晰度

**指标含义**：衡量 Issue 是否有明确的责任人（assignee）或认领者，用户能否知道"谁来负责这个问题"。不评价处理结果好坏，只评价责任是否清晰。

| 分数 | 判定条件 |
|------|---------|
| 100 | 明确 assignee + 候选负责人 + 下一步动作指定，责任完全清晰 |
| 80 | 明确 assignee，有候选负责人，但未指定下一步动作 |
| 60 | 有 assignee 但无候选负责人，或仅 bot 自动分配未经人工确认 |
| 40 | 无 assignee，但评论中有人表示愿意跟进（口头认领） |
| 20 | 无 assignee，但 Issue 有标签归类，隐含团队责任 |
| 0 | 无 assignee、无认领、无标签，完全无人负责 |

**不适用条件**：
- `obj-response-speed`：无社区评论 → not_applicable
- `obj-first-substantive-response-time`：无实质回应 → not_applicable
- 主观指标：无社区响应 → not_applicable

---

## 4. 阶段 3：讨论与解决（Discussion & Resolution）

评价从进入讨论到关闭的全过程。

- **起点**：进入实质讨论
- **终点**：Issue 关闭
- **核心问题**：问题在真正解决还是在空转？

**混合比例**：客观 60% + 主观 40%

### 指标

#### 7. `obj-solution-evidence` 解决证据强度

**指标含义**：衡量 Issue 的解决过程是否有可验证的依据支撑，如关联 PR、commit、文档、release notes、结果说明、用户确认等。评价"有没有证据"，不评价解决快慢。

| 分数 | 判定条件 |
|------|---------|
| 100 | PR(35)+commit(20)+文档(10)+release(10)+结果说明(15)+用户确认(10)，满分 |
| 80 | 有PR+commit+结果说明，缺文档/release/用户确认 |
| 60 | 有PR+结果说明，缺commit/文档/release/用户确认 |
| 40 | 有commit或文档+结果说明，无PR |
| 20 | 仅有结果说明，无PR/commit/文档 |
| 0 | 无任何证据 |

#### 8. `obj-result-formation-timeliness` 形成结果时效

**指标含义**：衡量从 Issue 创建到形成明确结果（关闭/PR合并/结论性评论/用户确认）需要多长时间。由 LLM 识别何时形成结果，再由程序计算时间差。

| 分数 | 判定条件 |
|------|---------|
| 100 | 形成结果 ≤3d |
| 80 | ≤7d |
| 60 | ≤14d |
| 40 | ≤30d |
| 20 | ≤60d |
| 0 | >60d 或未形成结果 |

#### 9. `sub-discussion-progression` 讨论推进性

**指标含义**：衡量讨论是否推动了问题前进，是否形成了下一步动作、结论或排查方向。区分"有效讨论"和"空转/灌水"，不评价最终是否解决。

| 分数 | 判定条件 |
|------|---------|
| 100 | 讨论形成明确结论+下一步+排查方向+PR/方案，完整推进闭环 |
| 80 | 讨论有明确下一步动作和排查方向，但尚未形成最终结论 |
| 60 | 讨论有基本方向，形成1-2个可执行下一步，但推进不完全 |
| 40 | 有讨论但停留在信息收集阶段，未形成可执行下一步 |
| 20 | 仅有1-2条无关评论，无实质讨论内容 |
| 0 | 无评论或评论纯为 +1/相同问题，零推进 |

#### 10. `sub-user-goal-result` 用户目标处理结果

**指标含义**：衡量用户提出的主要目标最终是否得到有效满足。不要求必须代码修复——修复、规避方案、正确答案、明确能力边界、合理拒绝、明确延期、有效转交均可获得中高分。

| 分数 | 判定条件 |
|------|---------|
| 100 | 用户主要目标完全满足（修复/合并PR + 用户确认解决） |
| 80 | 主要目标基本满足（PR已合并或给出有效规避方案），用户未表示不满 |
| 60 | 部分目标满足（给出临时方案/正确答案/明确能力边界），用户可接受 |
| 40 | 目标未满足但获得合理处置（明确拒绝/延期/转交其他仓库），有理由说明 |
| 20 | 目标未满足且处置不充分（仅回复"已知"但无后续计划） |
| 0 | 目标完全未回应，用户仍被阻塞，成熟期后仍无结果 |

**不适用条件**：
- `obj-solution-evidence`：未关闭且无PR/commit → not_applicable
- `obj-result-formation-timeliness`：未形成结果 → not_applicable
- `sub-discussion-progression`：评论数==0 → not_applicable
- `sub-user-goal-result`：未关闭且创建不满30天 → not_applicable

## 5. 阶段 4：总结与关闭（Closure）

评价关闭动作本身。

- **起点**：Issue 被关闭
- **终点**：关闭后确认稳定（或重开进入下一轮循环）
- **核心问题**：关得对不对？关了之后用户怎么办？

**混合比例**：客观 60% + 主观 40%

### 指标

#### 11. `obj-decision-transparency` 决策透明度

**指标含义**：衡量关闭 Issue 时是否清楚说明了关闭原因，让用户能理解"为什么关"。检查是否有 close_reason、关闭评论、是否静默关闭、wontfix/dup 说明、关闭总结是否充分。

| 分数 | 判定条件 |
|------|---------|
| 100 | 有close_reason(30)+关闭评论(25)+非静默关闭(20)+wontfix/dup说明(15)+总结>100字(10) |
| 80 | 有close_reason+关闭评论+非静默关闭，缺wontfix/dup说明或总结 |
| 60 | 有close_reason+关闭评论，但为静默关闭 |
| 40 | 有close_reason，但无关闭评论和总结 |
| 20 | 仅有close_reason，无其他说明 |
| 0 | 无close_reason、无关闭评论、静默关闭 |

#### 12. `obj-closure-reuse` 关闭复用价值

**指标含义**：衡量关闭后是否留下了可复用的知识或资产，如方案文档化、重复 Issue 的主链接、详细的关闭说明。评价"留了什么"，不评价关闭原因本身。

| 分数 | 判定条件 |
|------|---------|
| 100 | 方案文档化(30)+dup主链接(25)+关闭说明>50字(25)+关闭说明>100字(20)，满分 |
| 80 | 有方案文档化+dup主链接，但关闭说明不足 |
| 60 | 有方案文档化或dup主链接之一，关闭说明>50字 |
| 40 | 无方案文档化/dup主链接，但关闭说明>100字 |
| 20 | 关闭说明>50字但无文档化 |
| 0 | 无任何可复用内容 |

#### 13. `sub-closure-premature-close-risk-reverse` 过早关闭风险

**指标含义**：衡量 Issue 是否在未充分处理的情况下就被关闭。高分表示关闭时机合理、无过早风险；低分表示存在明显过早关闭迹象。反向分逻辑：高分=无风险。

| 分数 | 判定条件 |
|------|---------|
| 100 | PR已合并/方案已落地后关闭，关闭时机完全合理，无任何过早迹象 |
| 80 | 问题已解决但缺少用户确认即关闭，风险极低 |
| 60 | 有初步处理但用户未明确确认解决就关闭，存在轻微风险 |
| 40 | 处理不充分即关闭（如仅给建议未验证就关闭），有一定过早风险 |
| 20 | 未实质处理就关闭（如仅打标签就关闭），过早风险较高 |
| 0 | 创建后立即关闭/无任何处理就关闭，明显过早关闭 |

#### 14. `sub-followup-path-completeness` 后续路径完整性

**指标含义**：衡量关闭后是否告诉用户后续该怎么办，如重新开启条件、反馈入口、其他仓库/团队、后续版本、重新评估条件等。仅在关闭后仍可能需要反馈时评分。

| 分数 | 判定条件 |
|------|---------|
| 100 | 完整说明重新开启条件+反馈入口+其他仓库/团队+后续版本+重新评估条件 |
| 80 | 说明3项以上后续路径（如重新开启条件+反馈入口+后续版本） |
| 60 | 说明2项后续路径（如重新开启条件+反馈入口），但缺少版本/仓库信息 |
| 40 | 仅说明1项后续路径（如仅"如有问题可重新开启"），路径不完整 |
| 20 | 关闭说明中隐含后续可能但未明确任何路径 |
| 0 | 关闭后无任何后续路径说明，用户无法知道如何继续反馈 |

**不适用条件**：
- 全部指标：Issue 未关闭 → not_applicable
- `sub-followup-path-completeness`：问题已完整解决并获用户确认 → not_applicable

## 6. 阶段 5：自动化治理（Bot/Agent Governance）

评价 Bot 在全过程中的介入。

- **起点**：贯穿全程（Bot 可在任何阶段介入）
- **终点**：贯穿全程
- **核心问题**：自动化是在帮忙还是在添乱？

**混合比例**：客观 60% + 主观 40%

### 指标

#### 15. `obj-bot-governance` Bot治理有效性

**指标含义**：衡量 Bot/Agent 是否在 Issue 生命周期中做了有用的事，如 24h 内打标、追问信息、归并重复 Issue、关闭 stale Issue、触发人工跟进。无 Bot 介入时返回中性分 60。

| 分数 | 判定条件 |
|------|---------|
| 100 | 24h打标(20)+追问信息(15)+归并重复(20)+关闭stale(20)+人工跟进(25)，满分 |
| 80 | 完成4项有效动作，缺1项 |
| 60 | 完成3项有效动作，或无Bot介入（中性分） |
| 40 | 完成2项有效动作 |
| 20 | 仅完成1项动作 |
| 0 | Bot有介入但全部无效或制造噪音 |

#### 16. `obj-bot-misclose-reverse` Bot误关闭风险

**指标含义**：衡量 Bot 是否错误关闭了 Issue。风险分 = Bot 关闭后被重开 + maintainer 继续 + 添加产物；反向分 = 100 − 风险分。Bot 未关闭过 Issue 时返回 100（无风险）。

| 分数 | 判定条件 |
|------|---------|
| 100 | Bot未关闭过Issue（无风险），或关闭后无任何异常 |
| 80 | Bot关闭后有轻微异常但不影响处理（如添加产物） |
| 60 | Bot关闭后maintainer继续(30) |
| 40 | Bot关闭后被部分重开或maintainer继续 |
| 20 | Bot关闭后被重开(50) |
| 0 | Bot关闭后被重开+maintainer继续+添加产物，风险分满分 |

#### 17. `sub-bot-agent-helpfulness` 自动化帮助度

**指标含义**：衡量 Bot/Agent 的介入是否整体上对问题处理有帮助，而不是只制造噪音或做表面动作。守门型/权限型提示如果避免了错误操作也算帮助。

| 分数 | 判定条件 |
|------|---------|
| 100 | Bot有效打标+追问信息+归并重复+触发人工跟进，全流程帮助 |
| 80 | Bot完成3项有效动作（如打标+追问+分配），显著减少人工成本 |
| 60 | Bot完成2项有效动作（如打标+分配），有一定帮助但未全覆盖 |
| 40 | Bot仅完成1项动作（如仅打标），帮助有限 |
| 20 | Bot有介入但为模板回复，未触发有效处理，仅做表面动作 |
| 0 | 无Bot介入，或Bot制造噪音/误导/错误阻断 |

#### 18. `sub-bot-agent-intervention-quality` 介入动作质量

**指标含义**：衡量 Bot 的具体动作（打标签、提醒、引导、关闭等）是否准确、及时、合适。正确的权限校验、规则执行、流程守卫应给中高分，只有错误阻断或误判才低分。

| 分数 | 判定条件 |
|------|---------|
| 100 | 标签+提醒+引导+关闭全准确，时机合适，无任何错误动作 |
| 80 | 3项动作准确及时，1项次要动作不够理想（如标签延迟） |
| 60 | 2项动作准确（如标签+分配正确），但1项时机不佳或不够合适 |
| 40 | 动作基本正确但有1项明显失误（如标签打错后纠正） |
| 20 | 动作多为模板化，准确率低，需要人工多次纠正 |
| 0 | 动作错误/误导/错误阻断/误关闭，严重干扰处理流程 |

#### 19. `sub-bot-agent-human-handoff-quality` 人机交接质量

**指标含义**：衡量 Bot 动作后是否形成了良好的人类接手与推进。如果 Bot 介入后有人类继续承接，或 Issue 本身已有明确人工处理证据，不应低分。

| 分数 | 判定条件 |
|------|---------|
| 100 | Bot动作后人工立即接手，有明确 assignee + 后续推进，交接无缝 |
| 80 | Bot动作后人工较快速接手（24h内），有 assignee，交接顺畅 |
| 60 | Bot动作后人工接手但存在延迟（>24h），或 assignee 不够明确 |
| 40 | Bot动作后有人类响应但仅口头跟进，无实质性处理动作 |
| 20 | Bot动作后人工响应极其滞后（>7天），交接有明显停滞 |
| 0 | Bot动作后无人接手，流程被卡住，完全无人类承接 |

**不适用条件**：
- `obj-bot-governance`：无Bot介入 → 返回60（中性分）
- `obj-bot-misclose-reverse`：Bot未关闭过Issue → 返回100（无风险）
- 主观指标：无Bot介入 → not_applicable

## 8. 不适用处理原则

```text
if 指标 not_applicable:
    不计零分
    不进分母
    不影响总分

阶段分 = Σ（适用指标得分 × 权重）÷ Σ（适用指标权重）
```

---

## 9. 阶段分计算

### 9.1 客观分

```text
objective_score = Σ（客观指标得分 × 客观指标权重）÷ Σ（适用客观指标权重）
```

### 9.2 主观分

```text
subjective_score = Σ（主观指标得分 × 主观指标权重）÷ Σ（适用主观指标权重）
```

### 9.3 阶段分

```text
stage_score = objective_score × objective_blend + subjective_score × subjective_blend
```

### 9.4 痛点标记

```text
if stage_score < 60:
    tag = "痛点"
```

## 10. 混合客观指标说明

### 10.1 什么是混合客观指标

混合客观指标 = LLM 识别事实 + 程序计算分数。

LLM 的角色是**事实识别**（第二层语义事件），不是**质量判断**（第三层体验质量）。最终分数由固定算法计算，仍然是客观指标。

### 10.2 `obj-first-substantive-response-time` 首次实质回应时效

```text
第一步：LLM 识别
  输入：Issue 全部评论（按时间排序）
  LLM 任务：判断哪条评论是"首次实质回应"
  输出：comment_id + is_substantive + confidence

第二步：程序计算
  if is_substantive == false:
      score = 0
  else:
      first_substantive_response_hours = 评论时间 - 创建时间
      ≤48h→100, ≤4d→80, ≤7d→60, ≤14d→40, ≤30d→20, >30d→0
```

### 10.3 `obj-result-formation-timeliness` 形成结果时效

```text
第一步：LLM 识别
  输入：Issue 全部评论 + 关闭事件 + 关联产物
  LLM 任务：判断"何时形成明确结果"
  输出：result_formation_time + confidence

  结果形成信号优先级：
  1. Issue 已关闭 → closed_at
  2. 关联 PR 已合并 → pr_merged_at
  3. 评论中首次出现结论性内容 → 该评论时间
  4. 用户确认解决 → 确认评论时间
  5. None（未形成结果）

第二步：程序计算
  if result_formation_time is None:
      score = 0
  else:
      result_formation_hours = result_formation_time - 创建时间
      ≤3d→100, ≤7d→80, ≤14d→60, ≤30d→40, ≤60d→20, >60d→0
```

### 10.4 置信度处理

| 置信度 | 处理方式 |
|--------|---------|
| ≥0.80 | 正常计分 |
| 0.60～0.79 | 暂时计分，进入人工抽检池 |
| <0.60 | 不进入正式分数，标记待复核 |
